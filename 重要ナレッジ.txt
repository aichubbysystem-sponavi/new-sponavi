## 絶対ルール

### ログは要約して消さない
- 日々のログファイル（ログ/YYYY-MM/）は絶対に削除・上書き・要約しない
- 新しいログは別ファイルとして追加するのみ
- ハマったこと・デバッグ過程・試行錯誤の記録は後で必ず役に立つ
- 要約が必要な場合は、この重要ナレッジに追記する（元ログはそのまま残す）

## アーキテクチャの罠（最重要）

### Go API と Supabase は別世界
- 店舗データは2箇所に存在する:
  - Go API（Cloud Run）の PostgreSQL → /api/shop で返る店舗一覧
  - Supabase PostgreSQL → reviews, report系, 座標, カテゴリ等
- IDが異なる。Go APIのshop IDでSupabaseのshopsテーブルを検索しても見つからない
- 対策: sync-shopsのCronでGo API→Supabaseに毎日同期（2026-06-09追加）
- Supabaseのshopsを検索する時は name（店舗名）で検索すること、IDで検索しない

### ShopListItem.id は店舗名
- spreadsheet.ts:796 で `id: shopName` と設定されている
- レポート管理画面の selected Set にはUUIDではなく店舗名が入る
- report/[shopId] のURLパスも店舗名（encodeURIComponent済み）

### レポートの「対象月」は3箇所で独立管理される
- レポート管理画面: reportMonth ステート（対象月セレクタ）
- shop.period: スプレッドシートデータの最終月（"2026年4月"形式）
- curLabel: レポート表示側のmonthlyLabelsの最終要素（"2026/4"形式）
- メモ等を保存する時は reportMonth を優先すること（shop.periodだと月ズレする）

## よくあるバグパターン

### `.includes("")` は常に true
- 空文字をincludesに渡すとどんな文字列でもtrueになる
- 対策: `if (shopCity && shopAddress && fa.includes(shopCity) && fa.includes(shopAddress))`

### APIエラーを握りつぶさない
- Supabaseのテーブルが存在しない時、APIが `{ memo: "" }` を返す設計だと
  テーブル未作成に気づけない（report_memosで実際に発生）
- 開発中は `_error` フィールドを返すなどしてデバッグ可能にする

### デバウンスタイマーの共有問題
- 複数種類の保存を1つのタイマーrefで共有すると、
  異なる種類の連続トグルで前の保存がキャンセルされる
- 対策: 種類ごとに独立したタイマー（saveTimersRef.current[field]）

### detectLanguageはGBP原文を判定する（翻訳テキストではない）
- GBP口コミフォーマット: "(Translated by Google) [翻訳] (Original) [原文]"
- detectLanguage()は(Original)がある場合、その後ろ（原文）を判定対象にする
- 以前は(Original)以降を削除して翻訳テキストを判定していた → 韓国語口コミが日本語と誤判定されていた（2026-06-13修正）
- AI総評のanalyze/route.tsでもDB生コメントをdetectLanguageに渡す（fetchReviewsの加工済みcommentは使わない）

### Go APIの空フィールドでSupabaseを上書き
- sync時、Go APIの `state: ""` でSupabaseの `state: "北海道"` を消してしまう
- 対策: 値がある場合のみ updateRow に含める

## Next.js fetchキャッシュの罠（2026-06-16発見）
- Next.js 14のサーバーサイドfetch()はデフォルトでレスポンスをキャッシュする
- Google OAuthトークン交換もキャッシュされ、古い無効なアクセストークンが返り続ける
- 対策: 外部API呼び出しには必ず `cache: "no-store"` を付ける
- google-ads.ts のgetAccessToken()とexecuteGaql()に適用済み
