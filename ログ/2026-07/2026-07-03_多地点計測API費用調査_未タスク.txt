2026-07-03  多地点順位計測(グリッド) API費用の高騰 調査ログ
========================================================================

【発端】
Google Cloud「順位計測API」請求で 6月分 約17万円（Places API Text Search Pro ¥157,773＋税）。
さらに直近7日(6/27-7/3)で ¥417,628、うち 7/1単日で ¥261,662 のスパイク。
7/1の1日だけで6月まる1ヶ月を超えており、7月は6月の数倍ペースで悪化中。

【原因（コード確認済み・確定）】
費用の主犯は Places API (New) searchText = グリッド順位計測(多地点)。
座標設定・KW設定は主因ではない（KW設定は完全無料）。

1) 深さの問題（圏外店）
   - grid-ranking POST は 1地点につき最大5ページ探索（100位まで）
   - 圏外店は毎回5ページ全消費 = 1位表示店の5倍課金
   - 7×7=49地点なら 圏外店1KW=¥1,176 / 上位店1KW=¥235

2) 重複計測の問題（設計欠陥・最重要）★
   - 実測側(grid-ranking)に「当日計測済みならスキップ」のキャッシュ/ガードが無い
   - 一括計測ボタンを押すたびに、当日既に測った店・KWをゼロから測り直す
   - 実例: 「北海道海鮮居酒屋さぶろう 西8丁目店」7/2の履歴
       01:04-06 / 11:31-32 / 15:10-12 の3回、同一8KWを重複計測
       → 同日で順位はほぼ不変。2回目3回目は完全な捨て金（3倍払い）
       → さらに8KW中3つが101位=完全圏外（5ページ全消費・最高単価）
   - 「圏外 × 重複」のダブルパンチ。26万のうち推定2/3(約17万)が無駄と見られる

3) 物量の問題
   - 7/1のPlaces API ≒ 54,500リクエスト。今月計測済み194店 ≒ 1店280回
     = 平均4-5ページ = 圏外多発を裏付け

【ダッシュボードの爆弾（注意喚起）】
「未計測のみ一括計測（505店舗）」ボタン = 残り400店を全部計測。
現状のコードのまま押すと ¥30-50万規模の課金が一撃で発生。
→ コード修正が入るまで絶対に押さない。未計測店は放置なら¥0（課金されない）。
