========================================================================
2026-07-04  P-MAXレポート「まとめ」AI文章のキャッシュ実装
========================================================================

【背景】
/pmax/store の「まとめ」（Claude Sonnet 4.6生成、約¥4/回）が
レポートを開くたびに毎回生成されていた。
問題は費用（月¥2,000〜4,000）よりも「同じ月のレポートなのに
開くたびに文面が変わる」こと（顧客向けレポートとして不適切）。

【実装内容】
■ 新規テーブル: pmax_summary_cache（sql/2026-07-04_pmax_summary_cache.sql）
  shop_key / month('YYYY-MM') / kpi_hash / summary_text / unique(shop_key,month)

■ src/app/api/pmax/summary-text/route.ts
  1. KPIデータ+PROMPT_VERSIONのSHA-256ハッシュを計算
  2. キャッシュ照会: 同じ店×月×同じハッシュなら生成せず保存文面を返却
     { text, cached: true }
  3. ハッシュ不一致（データ再同期で数値が変わった）→ 再生成して上書き
     = 古い数値の文章が残る事故が構造的に起きない
  4. PROMPT_VERSION定数("v1"): テンプレート文言を改良したらここを上げる
     → 全店舗のキャッシュが自動無効化され新形式で再生成
  5. テーブル未作成/旧クライアント(shopName無し)は従来動作にフォールバック

■ src/app/pmax/store/page.tsx
  POST bodyに shopName / monthKey(curKey) を追加

【効果】
- 同じレポートの2回目以降の表示: Claude API ¥0 + 文面が完全固定
- 過去月レポート: データが変わらないため permanently キャッシュ
- 当月レポート: 同期でデータが変わった時だけ自動再生成（常に最新数値と整合）

【デプロイ手順】
1. Supabase SQL Editor で sql/2026-07-04_pmax_summary_cache.sql を実行
2. push済み → Vercel自動デプロイ
※ テーブル未作成でも壊れない（毎回生成の従来動作のまま）

【テスト観点】
- 同じ店舗のレポートを2回開く → 2回目は文面が同一・高速（cached:true）
- pmax_summary_cache に行が入る
- 全店舗を更新→当月データが変わった店だけ、次回表示時に文章が更新される
