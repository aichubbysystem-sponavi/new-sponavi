【背景】
同日の「多地点計測API費用調査」ログ参照。Places API Text Search Proが

ユーザー決定: ①キャッシュ+②共有のみ実装。③ページ数削減=保留。
④メインKW3×3化=見送り（従来通り メイン7×7実測 / サブ3×3実測+推定）。

【実装内容】
■ 新規テーブル: grid_search_cache（sql/2026-07-03_grid_search_cache.sql）
  keyword / lat_key / lng_key / month('YYYY-MM' JST) / places(jsonb 順位順店名配列)
  / complete(bool) / unique(keyword,lat_key,lng_key,month) / RLS有効(ポリシーなし=service roleのみ)

■ src/app/api/report/grid-ranking/route.ts POST 改修
  1. 計測地点を全国共通格子にスナップ: LAT_STEP=0.009° / LNG_STEP=0.011°(≈1km)
     → スナップ座標を「キャッシュキー」兼「実際のlocationBias中心」に使用（一貫性確保）
  2. キャッシュ照会: 同月×同KW×同格子点がヒットしたら API を呼ばず
     保存済みリストから店名マッチして返却 { rank, cached: true }
     - rank発見 or complete=true → キャッシュで確定
     - 不完全リストで未発見 → 実測にフォールバック（リストを完全版に更新）
  3. 実測時: 全ページの店名を allPlaces に蓄積し、リスト全体を upsert 保存
     - 従来は自店の順位だけ読んでリストを捨てていた（1回¥4.8の情報を使い捨て）
     - complete判定: 最終ページ到達(次トークンなし/5ページ)=true、途中発見打ち切り=false
  4. findRank(): 店名マッチを関数化。空文字ガード追加
     （placeName="" だと targetName.includes("") が常にtrueになる罠 → ナレッジ通り）

【効果（設計上）】
- 同月内の再計測（同じ店・同じKW・同じ地点）→ API呼び出しゼロ
  例: 7/2に3回計測された店 → 2回目以降¥0（従来は毎回全額）
- 近隣店舗が同KWを追跡 → 格子共有により1回の検索で複数店舗の順位確定
- 後から店舗追加しても、その月の計測済みリストから遡って順位取得可能

【トレードオフ（ユーザー周知済み想定）】
- 計測中心が格子スナップで最大約700m（typ. 350m）ズレる。
  locationBias半径2000mに対して影響軽微。表示グリッド座標は従来のまま
- キャッシュは月単位 → 同月内は順位が固定される（月次レポート用途では問題なし）
- 途中発見打ち切りの不完全リストでは、下位店舗の照会時に再実測が走る（従来と同額）
