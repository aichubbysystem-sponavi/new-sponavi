# 開発ルール

## 修正・実装の手順（必ずこの順番で進めること）

### 1. 現状把握（コードを触る前に必ずやる）
- 関連するログファイル（ログ.txt）を最初に読む
- エラーが起きている箇所だけでなく、**実際のデータ**（スプレッドシートの中身、APIレスポンス、DBの値など）を確認する
- 「このデータがこういう形で来たらどうなるか」を想定してからコードを読む

### 2. 影響範囲の全体把握（修正箇所を決める前にやる）
- 修正対象の関数・APIを**呼び出している側**（フロントエンド、cron、他のAPI）を全て特定する
- データの流れを上流→下流まで追う（スプレッドシート → API → フロント表示）
- 「この値がプレビュー時と実行時で変わらないか」「件数が途中で増減しないか」を確認する

### 3. 実装（最小限かつ確実に）
- 修正は1箇所で終わらせず、同じパターンのバグが他にないか検索する（予約投稿、cron実行など）
- フォールバック・代替処理を入れる場合、既存の正常フローを壊さないことを確認する

### 4. セルフレビュー（コミット前に必ずやる）
以下の観点で自分のコードを検証する：
- **データ不一致**: プレビューと実行で件数・形式が異なるケースはないか
- **ループ条件**: 展開・フィルタで件数が変わった場合、ループが正しく回るか
- **部分一致の罠**: 名前マッチで意図しない店舗にマッチしないか
- **キャッシュ**: リクエスト間でステートが残って汚染しないか
- **タイムアウト**: 外部API呼び出しが増えた場合、全体の実行時間は制限内か

### 5. ビルド確認 → デプロイ → 動作報告
- `next build` でエラーがないことを確認してからコミット
- デプロイ後、ユーザーに「何をテストすべきか」「期待される結果」を具体的に伝える

## やってはいけないこと
- コードだけ見て推測で修正する（実データを必ず確認）
- 修正箇所だけ見て「完了」と報告する（呼び出し元・関連画面まで確認）
- 「多分大丈夫」で済ませる（大丈夫でない理由を探してから報告）

## プロジェクト固有のルール
- セッション終了時は必ずログに作業内容を追記する（ログ/YYYY-MM/ フォルダに日付付きファイルで保存）
- 新セッション開始時に以下の両方を必ず読むこと：
  1. ログ.txt（末尾100行程度 — 過去の経緯と移行案内）
  2. ログ/ フォルダ内の最新ファイル（直近セッションの詳細）
